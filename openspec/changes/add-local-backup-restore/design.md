## Context

当前 App 是纯本地记账应用，账单数据存储在 Room 数据库中，不依赖登录、云同步或服务端接口。用户换手机时，旧设备中的本地账单无法自动迁移到新设备，需要通过本地文件完成导出和导入。

设置页已经承担本地数据管理职责，本次功能继续放在设置页，新增“生成数据库数据”和“读取数据库数据”两个入口。

## Goals / Non-Goals

**Goals:**

- 旧手机可以把当前账单数据导出成一个本地备份文件。
- 新手机可以把备份文件放到指定目录后，在 App 内读取并恢复账单数据。
- 导入完成后 App 内明细、统计等页面使用恢复后的数据库数据。
- 全流程保持纯本地，不请求网络。
- 备份文件格式具备版本号，方便后续字段扩展。

**Non-Goals:**

- 不做账号体系、云同步、局域网传输或二维码传输。
- 不做跨 App 自动文件发送。
- 不导出系统设置、主题、缓存或非账单业务数据。
- 不把数据库文件本体直接暴露为备份格式，避免 Room 版本和设备路径耦合。

## Decisions

### 备份格式使用 JSON

导出文件使用 JSON，而不是直接复制 SQLite 数据库文件。

原因：
- JSON 可以包含 `version`、`exportedAt`、`records` 等元数据。
- 后续 Room 表字段变化时，可以通过备份版本做兼容解析。
- 比直接复制数据库文件更容易做去重、合并和错误提示。

建议文件结构：

```json
{
  "version": 1,
  "exportedAt": 1720000000000,
  "records": [
    {
      "id": 1,
      "type": "EXPENSE",
      "category": 1,
      "amountCents": 1200,
      "note": "午餐",
      "occurredAt": 1720000000000,
      "createdAt": 1720000000000,
      "updatedAt": 1720000000000
    }
  ]
}
```

### 备份目录使用 App 外部文件目录

导出文件放在 App 可访问的外部文件目录，例如：

`Android/data/com.example.keepaccount/files/backup/keep_account_backup.json`

原因：
- App 可以直接读写该目录，不需要用户手动授权复杂权限。
- 用户可以通过文件管理器或数据线把文件复制到新手机相同目录。
- 目录明确，便于设置页提示用户。

### 导入策略默认合并导入

导入备份文件时默认合并到现有数据库，而不是先清空数据库。

去重建议：
- 优先使用备份记录中的原始 `id` 作为参考，但不直接强依赖跨设备 id。
- 使用 `type + category + amountCents + occurredAt + note + createdAt` 作为内容去重键。
- 已存在的记录跳过，不存在的记录插入。

原因：
- 新手机可能已经有少量新数据，合并导入更安全。
- 清空数据库属于高风险操作，应避免作为默认恢复策略。

### 导入后刷新 App 状态，不强制重启

导入完成后，Repository/ViewModel 应重新查询当前月份、统计数据和设置状态。默认不要求用户重启 App。

如果实现中发现某些缓存或页面状态无法可靠刷新，才显示“导入完成，请重启 App 生效”的明确提示。

## Risks / Trade-offs

- [Risk] 用户把文件放错目录导致读取失败 → 在设置页提示完整目录和文件名，并在失败时显示原因。
- [Risk] JSON 文件损坏或版本不支持 → 校验 `version` 和 `records`，失败时不写入数据库。
- [Risk] 导入重复数据 → 使用内容去重键跳过重复记录。
- [Risk] 导入过程中部分成功部分失败 → 解析成功后再批量写入；写入阶段尽量使用事务。
- [Risk] 备份文件包含未来版本字段 → 版本 1 先忽略未知字段；未来不兼容版本给出提示。
